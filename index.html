Test-driven development
Tesztelés a fejlesztés részeként

A szoftvertesztelés a szoftverminőség-biztosítás és így a szoftverfejlesztés részét képezi. A tesztelés egy rendszer vagy program kontrollált körülmények melletti futtatása, és az eredmények kiértékelése. A hagyományos megközelítés szerint a tesztelés célja az, hogy a fejlesztés során létrejövő hibákat minél korábban felfedezze, és ezzel csökkentse azok kijavításának költségeit.

Jelenleg a tesztelői munka egyre inkább eltolódik a fejlesztők és a döntéshozók információkkal való támogatásának irányába. Fontos szerepek még a fejlesztés állapotának pontos és egzakt mérése és a kockázatkezelés és -becslés.

Egységtesztelés

A számítógép-programozásban az egységtesztelés a szoftvertesztelésnek egy olyan módszere, amelynek során a forráskód egységeit (egy vagy több számítógépes program modul készletet) a kapcsolódó vezérlő adatokkal, a felhasználási-és a működtető eljárásokkal együtt tesztelik annak meghatározására, hogy azok elérik-e kitűzött céljukat.

Alapvetően fejlesztők végzik, hogy eldöntsék azt, hogy kódjuk megfelelően működik-e és megfelel a felhasználói előírásoknak. Kipróbálják az általuk írt kódrészleteket, például osztályokat, funkciókat, interfészeket és eljárásokat. A szoftver hibáinak észlelése a "nap elején" annak előnye, hogy ezzel a csapat minimalizálja a szoftverfejlesztési kockázatokat, mert az időt és pénzt pazarolja, ha vissza kell térnie és meg kell oldania a program alapvető problémáit, miután majdnem elkészült.

A unit-teszt, vagy más néven egységteszt, a metódusokat teszteli. Adott paraméterekre adjuk meg a metódus visszatérési értékét. A unit-teszt megvizsgálja, hogy a tényleges visszatérési érték megegyezik-e az elvárttal. Ha igen, sikeres a teszt, egyébként sikertelen.

Az egységtesztelés lépései

  1. Értelmezési tartomány meghatározása

    2. Milyen adatokra van értelmezve a feladat..

  2. Értékkészlet meghatározása

    3. Ezeket adhatja vissza a metódus..

  3. Input-Output értékpárok meghatározása

    4. Adott bemenő érték esetén mi a kimeneti érték?

  4. Teszt megírása

    5. Ez a tesztesetek elkészítését jelenti. (JUNIT5)

  5. Implementációs kód elkészítése

    6. Metódusunk elkészítése..

  6. Teszt futtatása

    7. FAILED, egy teszteset vagy a függvény hibás..

    PASSED, új teszteseteket kell írni.

  7. Refaktorálás

    8. Ez a folyamat a forráskód elemzését jelenti.

  8. Dokumentáció elkészítése

    9. Fejlesztőknek dokumentáljuk a tesztpéldányunkat.

Felhasználható szoftverek

IntelliJ

Az IntelliJ IDEA egy Java-ban írt fejlesztői környezet a JetBrains-től. Hasznos alapfunkciói közül legjobbnak a kódkiegészítés, refaktorálás vagy a debuggolás mondható.

Letölthető itt

JUNIT5

A JUNIT az egyik legnépszerűbb egységteszteléshez használt keretrendszer a Java ökoszisztémájában. A JUnit 5. verziója jópár érdekes újdonságot vezetett be, azzal a céllal, hogy a Java 8 vagy újabb verzióinak több hasznos funkcióját is támogassa.

Maven, függőségek

Az Apache Maven egy szoftver, amelyet szoftverprojektek menedzselésére és a build folyamat automatizálására lehet használni. Funkcionalitásában hasonlít az Apache Ant eszközhöz.

A Maven bevezeti a POM, azaz a Projekt Objektummodell (angolul: Project Object Model) fogalmát. Egy POM egy buildelendő projektet ír le és annak függőségeit.

Forrás: Wikipédia

A JUnit 5.x.x hozzáadása igen egyszerű, szimplán hozzáadjuk az alábbi három függőséget az automatikusan generált pom.xml-fájlunkhoz IntelliJ alatt. (dependencies tagek közé!) 


<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter-api</artifactId>
    <version>5.7.0</version>
    <scope>test</scope>
</dependency>

<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter-engine</artifactId>
    <version>5.7.0</version>
    <scope>test</scope>
</dependency>

<dependency>
    <groupId>org.junit.jupiter</groupId>
    <artifactId>junit-jupiter-params</artifactId>
    <version>5.6.0</version>
    <scope>test</scope>
</dependency>

Gyorsbillentyűk, hotkeys

Az IntelliJ IDEA gyorsbillentyűparancsokkal, másnéven "hotkey"-ekkel rendelkezik a parancsok többségéhez, amelyek a szerkesztéshez, a navigációhoz, a refaktoráláshoz, a hibakereséshez és más feladatokhoz kapcsolódnak. Ezeknek a gyorsbillentyűknek a használata segít hozzá a gyorsabb munkavégzésben.

Shift+F9

Ezzel lehet hibakeresési módban futtatni az Intellij projektet.

ALT+ENTER

Elem classpath importálása..

Ctrl+Y

Egy egész sort töröl.

Ctrl+Shift+A

Parancs keresése és futtatása...

Ctrl+W, Ctrl+Shift+W

Kijelölés bővítése, csökkentése..

Ctrl+Alt+S

Billentyűtérkép a beállítások párbeszédpanelen belül..

Ctrl+E

Mostanában megnyitott fájlok..

Alt+Insert

Előre legenerálható kódok..

Ctrl+P

Ezzel a billentyűzetparanccsal lehet a már bezárt ablakokat újból meghívni.

Shift+F6

Ezzel a billentyűzetparanccsal lehet átnevezni a kódban a fájlok, könyvtárak, csomagok, modulok nevét és az azokra vonatkozó összes hivatkozást.

Shift+Shift

Keresés mindenhol az intelliJ-ben. Gyorsan megtalálhat bármilyen fájlt, műveletet, szimbólumot, eszközablakot vagy beállítást a projektben és az aktuális Git-tárban..

Ctrl+D

Duplikálja, azaz pontos másolatot készít a már kijelölt sor vagy kijelölt elemről.

Mintafeladatok

Készítsünk tesztet egy egyszerű átváltó metódusra.. (a kilométer és méter ismeretével méterbe..)
public class TDDProjekmukaEgyikFeladat {
    @Test
    @DisplayName("Első feladat")
    public void elsoFeladatTest() {

        assertEquals(-1, kiSzamol(-64, -10));
        assertEquals(0, kiSzamol(0, 0));
        assertEquals(10256, kiSzamol(10, 256));
        assertEquals(-1, kiSzamol(45, -5));
        assertEquals(-1, kiSzamol(-5, 45));
        assertEquals(36230, kiSzamol(35, 1230));
        assertEquals(-1, kiSzamol(-3, 1003));
        assertEquals(6000, kiSzamol(6, 0));
        assertEquals(124, kiSzamol(0, 124));

}
    
    private int kiSzamol(int km, int m) {
           if (km < 0 || m < 0 ||)
               return -1;
        return 1000*km+m;
     }
}

Értékkészlet:

Óra: 0-23

Perc: 0-59

Kategóriák meghatározása:

  • mindkettő negatív
  • mindkettő nulla
  • mindkettő jó
  • egyik nulla, másik negatív
  • egyik nulla, másik túlcsordult
  • egyik jó, másik negatív
  • egyik jó, másik túlcsordult

Ha hibás értéket adunk át paraméterként, akkor térjen vissza -1-el, ha túlcsordultat, -2-vel.

public class TDDProjekmukaMasikFeladat {
    @Test
    @DisplayName("Második feladat")
    public void masodikFeladatTest() {

        assertEquals(-1,toPerc(-2,-4));
        assertEquals(0,toPerc(0,0));
        assertEquals(-1,toPerc(-2,0));
        assertEquals(-1,toPerc(0,-4));
        assertEquals(-2,toPerc(24,0));
        assertEquals(-2,toPerc(0,61));
        assertEquals(-1,toPerc(2,63));
        assertEquals(-1,toPerc(30,2));
        assertEquals(-1,toPerc(-2,4));
        assertEquals(-1,toPerc(2,-4));
        assertEquals(694,toPerc(11,34));


}
    private int toPerc(int ora, int perc){
        if (ora < 0 || perc < 0)
            return -1;
        if (ora > 23 || perc > 59)
            return -2;
        return ora * 60 + perc;
    }
}
